# Требования к MVP инициатив CVM Agents / AdTarget

> Цель документа — зафиксировать MVP-требования по всем инициативам из клиентского shortlist, опираясь на текущие возможности репозитория `cvm-agents` и заметки по UCell, СберМобайл, ТТ Мобайл, Tele2 Казахстан и KCell.

## 1. Подход к формированию MVP

MVP описан не как целевая «большая платформа», а как минимальный демонстрационный и пилотный объем по каждой инициативе:

1. Если в репозитории уже есть готовая основа, требования ограничиваются тем, что можно показать с минимальной доработкой.
2. Если по инициативе готовой реализации нет, MVP формулируется по клиентским заметкам: что именно нужно показать/проверить в пилоте, без лишних функций из соседних инициатив.
3. Требования не смешивают инициативы между собой: например, мониторинг не включает полноценную оптимизацию, а Copilot не включает создание кампаний.

### 1.1. Что уже есть в репозитории

Текущий прототип уже содержит основу для части инициатив:

- **Campaign Builder** — чатовый агент для генерации кампании из текстовой цели, сборки flow, уточнения параметров, использования справочников AdTarget и создания кампании.
- **CVM Copilot** — чат с RAG по документации и контекстом текущего экрана/кампании.
- **Campaign Monitor** — панель с оценкой кампании, демо-метриками, рекомендациями, funnel, каналами доставки и test/control сравнением.
- **AdTarget mock UI** — визуальный mock редактора flow и плавающий AI-widget с вкладками Copilot / Builder / Monitoring.
- **Mock/контурный режим** — `ADTARGET_MOCK=true` и fallback на mock-данные при недоступности AdTarget API.
- **Справочники и flow helpers** — mock/AdTarget справочники ЦГ, каналов, событий, offer templates, типов и групп кампаний; генерация communication activity и business transaction activity.

### 1.2. Список инициатив в scope документа

В документе описаны MVP-требования для 7 инициатив из shortlist:

1. **Агент проектирования кампании**.
2. **Эмулятор кампаний / Синтетическая песочница**.
3. **Агент мониторинга и оценки кампаний**.
4. **Интеллектуальный помощник для CVM-команды / CVM-copilot**.
5. **Агент оптимизации кампаний**.
6. **Генерация контента**.
7. **Автоматическая сегментация**.

## 2. Агент проектирования кампании

### 2.1. Что прототипируем

Чат и окно генерации, в котором пользователь описывает идею кампании, а агент собирает один из нескольких сценариев и создает кампанию в самом AdTarget или mock-AdTarget.

### 2.2. Клиентские ожидания из заметок

- **UCell**: нужен помощник, который комплексно видит продукт, кампанию, сегмент и параметры; может рекомендовать изменения до согласования.
- **СберМобайл**: агент должен уточнять структуру кампании после первичной сборки, задавать вопросы и давать рекомендации по проектированию.
- **Tele2 Казахстан**: нужен чатовый интерфейс, где продуктолог/маркетолог описывает идею, а система собирает структуру кампании для проверки гипотезы.
- **KCell**: интересна генерация кампаний и развитие AdTarget как low-code среды для агентских сценариев.

### 2.3. Опора на готовое

Уже есть `POST /api/builder`, backend-сессии Builder, чатовая UI-панель, справочники AdTarget/mock, функции сборки flow, auto-create кампании и отображение generated flow на mock-canvas.

### 2.4. MVP-требования

1. Пользователь открывает окно генерации кампании из AI-widget и вводит цель кампании в свободной форме.
2. Агент поддерживает минимум 2–3 демонстрационных сценария генерации:
   - простая коммуникационная кампания `Common → TargetGroup → SMS/Push/Email`;
   - event-triggered кампания `Common → TargetGroup → Event → SMS/Push`;
   - продуктовая кампания `Common → TargetGroup → SMS/Push → BusinessTransaction`.
3. Агент использует только доступные справочники AdTarget/mock: целевые группы, каналы, события, offer templates, campaign types/groups.
4. Если не хватает обязательных параметров, агент задает уточняющий вопрос в чате.
5. После генерации пользователь видит flow в редакторе AdTarget mock.
6. Агент создает кампанию в AdTarget/mock и возвращает `campaign_id`, если создание успешно.
7. Пользователь может follow-up сообщением изменить текущую кампанию: канал, текст, offer, событие или простую activity.
8. Запуск кампании остается подтверждаемым действием пользователя, а не полностью автономным действием агента.

### 2.5. Границы MVP

Входит:

- чат и окно генерации;
- 2–3 стабильных сценария для демо;
- создание кампании в AdTarget/mock;
- отображение результата во flow;
- уточняющие вопросы и follow-up доработка.

Не входит:

- покрытие всех типов activity AdTarget;
- сложные ветвления и полноценный low-code designer агентов;
- approval-процесс и роли;
- генерация новых сегментов — это отдельная инициатива «Автоматическая сегментация».

### 2.6. Acceptance criteria

- По заранее согласованным demo-prompts агент создает кампанию и показывает ее во flow.
- Если параметр не определен, агент спрашивает, а не выдумывает справочник.
- После создания виден `campaign_id` и текущий `draft_flow`.
- Follow-up в той же сессии меняет текущую кампанию.

## 3. Эмулятор кампаний / Синтетическая песочница

### 3.1. Что прототипируем

Реестр тестовых кампаний и тестирование кампаний до боевого запуска: безопасный test run на draft/mock-данных с отображением ошибок и результата проверки.

### 3.2. Клиентские ожидания из заметок

- **UCell**: тестовые кампании сейчас мешают боевым, портят статистику; нужен отдельный тестовый контур, иногда достаточно протестировать механику, продукт, бизнес-транзакцию или канал.
- **СберМобайл**: нужен тестовый прогон кампании перед заведением боевой, с подсветкой некорректных блоков/структур и рекомендациями.
- **ТТ Мобайл**: нужен эмулятор или реестр тестовых кампаний; ошибки должны отмечаться на flow с объяснением проблем по ветке коммуникации.
- **KCell**: эмулятор интересен как проверка корректности и оптимальности кампаний перед запуском.

### 3.3. Опора на готовое

Частично готово: есть mock-режим, Builder sessions, сохранение `draft_flow`, pre-launch Monitoring, отображение flow и ошибок на mock UI. Отдельного реестра тестовых кампаний пока нет.

### 3.4. MVP-требования

1. Система предоставляет режим «Тестовая кампания / Песочница» для кампаний, созданных Builder или выбранных из истории.
2. Пользователь видит реестр тестовых кампаний с минимумом полей:
   - название/цель;
   - `campaign_id` или временный id;
   - дата создания/обновления;
   - статус теста: draft, test passed, has issues;
   - краткое число ошибок/предупреждений.
3. Пользователь запускает тестовый прогон без боевого старта кампании.
4. Test run проверяет минимальный набор правил:
   - выбрана целевая группа;
   - есть активная коммуникация или бизнес-транзакция;
   - заполнен текст коммуникации;
   - корректно выбран канал для типа контента;
   - business transaction имеет offer template / operation;
   - нет очевидно пустой ветки.
5. Результат теста отображается в виде списка ошибок/предупреждений и рекомендаций.
6. Если UI позволяет, проблемные activity подсвечиваются на flow; если нет — ошибка должна содержать имя/id activity.
7. Тестовый прогон не меняет боевую статистику и по умолчанию работает на draft/mock-данных.

### 3.5. Границы MVP

Входит:

- простой реестр тестовых кампаний на базе Builder sessions/campaign states;
- ручной запуск test run;
- структурные проверки draft flow;
- отображение ошибок и рекомендаций;
- явная маркировка synthetic/mock режима.

Не входит:

- полноценная синтетическая база абонентов;
- реальное выполнение бизнес-транзакций в биллинге;
- интеграционные тесты внешних каналов;
- нагрузочное тестирование;
- отдельная production-изоляция статистики в DWH.

### 3.6. Acceptance criteria

- Пользователь видит список тестовых кампаний.
- Для выбранной кампании можно запустить test run без боевого запуска.
- Результат показывает ошибки, warnings и рекомендации.
- Ошибки привязаны к activity или ветке настолько детально, насколько доступно в текущем flow JSON.

## 4. Агент мониторинга и оценки кампаний

### 4.1. Что прототипируем

Чат/панель анализа кампании, отображение детализированных ошибок и рекомендаций, а также выдача информации по данным после запуска.

### 4.2. Клиентские ожидания из заметок

- **ТТ Мобайл**: оптимизация видится как выявление уже работающих некорректных кампаний и тестовый прогон готовящихся; нужно объяснять проблемы в конкретных ветках коммуникации.
- **Tele2 Казахстан**: агент должен находить старые, неработающие или некорректные кампании, которые не набирают аудиторию или содержат ошибки в бизнес-транзакциях.
- **KCell**: интересен технический мониторинг кампаний, проверка корректности и оптимальности.
- **UCell**: нужен отчет перед согласованием и рекомендации по готовой кампании.

### 4.3. Опора на готовое

Уже есть `POST /api/monitor`, панель Monitoring, расчет демо-KPI, score, summary, recommendations, funnel, channel delivery и test/control comparison. Чат мониторинга как отдельный интерфейс пока не реализован.

### 4.4. MVP-требования

1. Пользователь открывает агента мониторинга для конкретной кампании.
2. Агент показывает краткое заключение и score кампании.
3. Агент показывает детализированные ошибки и рекомендации:
   - структурные ошибки flow;
   - проблемы с целевой группой;
   - проблемы с коммуникацией;
   - проблемы с business transaction;
   - рекомендации по исправлению.
4. Для запущенной кампании агент показывает данные после запуска:
   - отправлено;
   - доставлено;
   - открыто/прочитано, если применимо;
   - клики, если применимо;
   - активации/целевые действия;
   - conversion rate;
   - сравнение test/control при наличии.
5. Агент должен различать pre-launch и post-launch режимы: до запуска он не должен выдавать synthetic metrics за реальные.
6. Пользователь может обновить анализ вручную.
7. MVP-чат мониторинга должен поддерживать минимум вопросы: «что не так?», «что исправить перед запуском?», «почему низкая конверсия?», «какие кампании похожи?».

### 4.5. Границы MVP

Входит:

- анализ одной выбранной кампании;
- детализированные ошибки и рекомендации;
- post-launch summary по доступным/демо-данным;
- ручное обновление анализа;
- простой чат поверх результатов анализа или быстрые вопросы в панели.

Не входит:

- инвентаризация тысяч legacy campaigns;
- автоматическая остановка или изменение кампаний;
- полноценное подключение DWH/BI;
- оптимизация параметров кампании — это отдельная инициатива «Агент оптимизации кампаний».

### 4.6. Acceptance criteria

- Для выбранной кампании показаны score, summary, ошибки и рекомендации.
- До запуска интерфейс явно показывает pre-launch режим.
- После запуска отображаются post-launch показатели.
- Пользователь может задать вопрос по результатам анализа и получить объяснение.

## 5. Интеллектуальный помощник для CVM-команды / CVM-copilot

### 5.1. Что прототипируем

Чат для диалогов по документации AdTarget, CVM-процессам и общим вопросам команды CVM.

### 5.2. Клиентские ожидания из заметок

- **UCell**: нужен помощник, который снижает сложность работы с кампаниями и данными, объясняет рекомендации и помогает готовить запуск.
- **KCell**: интересен общий агентский слой вокруг AdTarget.
- Для остальных операторов Copilot полезен как интерфейс объяснения документации, ошибок и CVM-процессов, но не должен подменять Builder/Monitor.

### 5.3. Опора на готовое

Уже есть `POST /api/copilot`, RAG по документации, контекст текущего экрана/кампании, ChatPanel, история диалога и citations.

### 5.4. MVP-требования

1. Пользователь задает вопрос в чате Copilot.
2. Copilot отвечает по документации AdTarget и общим CVM-вопросам.
3. Copilot учитывает текущий экран, роль пользователя и `campaign_id`, если они переданы.
4. Ответы по документации сопровождаются источниками/citations.
5. Copilot объясняет типовые ошибки flow и валидации человеческим языком.
6. Copilot предлагает следующий шаг, но не меняет кампанию сам.
7. Если информации в документации нет, Copilot должен явно сказать, что данных недостаточно.

### 5.5. Границы MVP

Входит:

- чат Q&A по документации;
- объяснение CVM/AdTarget терминов и ошибок;
- учет текущего экрана и кампании;
- ссылки на источники.

Не входит:

- создание кампаний — это инициатива «Агент проектирования кампании»;
- мониторинг кампаний — это инициатива «Агент мониторинга и оценки кампаний»;
- оптимизация параметров — это инициатива «Агент оптимизации кампаний»;
- корпоративная база знаний клиента, если документы не загружены.

### 5.6. Acceptance criteria

- Copilot отвечает на вопросы по документации с источниками.
- Copilot объясняет ошибку текущей кампании и следующий шаг.
- Copilot не обещает действий, которых не выполняет.
- При отсутствии данных Copilot корректно сообщает ограничение.

## 6. Агент оптимизации кампаний

### 6.1. Что прототипируем

Минимальный агент рекомендаций по улучшению параметров кампании. В текущем репозитории отдельной реализации нет, поэтому MVP формируется по клиентским заметкам и может быть реализован как новый слой рекомендаций поверх campaign flow, справочников и результатов мониторинга.

### 6.2. Клиентские ожидания из заметок

- **UCell**: не хватает оптимизации параметров кампании — времени контакта, канала, продукта и других настроек; текущие бизнес-правила и ML-модели долго согласуются и не дают комплексного взгляда продукт + кампания + сегмент.
- **KCell**: интересен агент оптимизации кампаний вместе с мониторингом.
- **Tele2 Казахстан**: под оптимизацией важен технический анализ запущенных кампаний и выявление legacy/некорректных сценариев; при этом решение об остановке/запуске должно оставаться за человеком.

### 6.3. Опора на готовое

Готовой отдельной реализации нет. Можно переиспользовать данные Builder/Monitor: `draft_flow`, campaign status, generated offers, channel metrics, recommendations и справочники каналов/offer templates.

### 6.4. MVP-требования

1. Агент анализирует одну выбранную кампанию или draft flow.
2. Агент выдает рекомендации по параметрам кампании, не применяя их автоматически:
   - канал коммуникации;
   - время/окно контакта как текстовая рекомендация;
   - продукт/offer template;
   - наличие control/test group;
   - текст коммуникации, если он влияет на эффективность.
3. Каждая рекомендация должна содержать:
   - что изменить;
   - почему это может улучшить результат;
   - ожидаемый эффект качественно: снизит риск, повысит доставку, повысит конверсию, упростит flow;
   - уровень уверенности: low/medium/high.
4. Агент должен различать рекомендации до запуска и после запуска.
5. После запуска агент использует доступные метрики мониторинга для объяснения рекомендаций.
6. Пользователь подтверждает применение рекомендаций вручную; MVP не меняет кампанию автоматически.

### 6.5. Границы MVP

Входит:

- рекомендационный отчет по одной кампании;
- оптимизация на уровне советов, а не автоматического применения;
- использование текущего flow и результатов мониторинга;
- человек принимает финальное решение.

Не входит:

- обучение ML-моделей;
- предиктивный выбор лучшего времени/канала на реальных исторических данных;
- автоматическое A/B/n и champion-challenger;
- автоматическая остановка или изменение кампаний;
- расчет выручки и маржинальности.

### 6.6. Acceptance criteria

- Для выбранной кампании агент формирует 3–5 рекомендаций по оптимизации.
- Рекомендации разделены на pre-launch и post-launch, если применимо.
- Каждая рекомендация объясняет причину и ожидаемый эффект.
- Ни одна рекомендация не применяется без подтверждения пользователя.

## 7. Генерация контента

### 7.1. Что прототипируем

Предварительный показ персонализированной генерации текста/оффера по продуктовому каталогу прямо во flow кампании.

### 7.2. Клиентские ожидания из заметок

- **Tele2 Казахстан**: наиболее актуальна гиперперсонализация и генерация контента: сообщения, креативы и офферы на основе продукта, сезона и профиля пользователя; интересен A/B-подход с изменением формулировок.
- **UCell**: интересна генерация контента и подбор предложений с помощью ИИ в защищенном контуре.
- **СберМобайл**: интересны рекомендации продуктов/NBA на основе LLM и свои inference/контурные модели.
- **KCell**: уже используют агентские сценарии для генерации, например IVR-скриптов; интересна генерация под каналы коммуникации.

### 7.3. Опора на готовое

Частично готово: Builder хранит preferences по продукту/контенту, flow builder создает communication activity с текстом, mock offer templates доступны через справочник, generated offers извлекаются из flow. Отдельного продуктового каталога и полноценной персонализации пока нет.

### 7.4. MVP-требования

1. Пользователь в Builder выбирает или описывает продукт/оффер из доступного каталога/справочника.
2. Агент генерирует текст коммуникации для выбранного канала: SMS, Push или Email.
3. Генерация учитывает минимум:
   - продукт/offer;
   - канал;
   - целевой сегмент или архетип;
   - тональность: нейтральная, промо, сервисная;
   - сезонный/контекстный повод, если пользователь его указал.
4. Сгенерированный текст отображается во flow в communication activity.
5. Если выбран продуктовый сценарий, агент подбирает существующий offer template/business operation из справочника.
6. Пользователь может попросить 2–3 варианта текста и выбрать один.
7. MVP демонстрирует персонализацию на уровне сегмента/архетипа, а не индивидуального абонента.

### 7.5. Границы MVP

Входит:

- генерация текста для SMS/Push/Email;
- подбор оффера из существующего справочника;
- отображение результата во flow;
- 2–3 варианта текста для сравнения;
- защищенный контур в demo-смысле: без реальных персональных данных абонентов.

Не входит:

- генерация изображений, видео или reels;
- уникальный текст для каждого абонента;
- полноценный NBA на реальных данных;
- автоматическая ротация вариантов каждые 2–3 дня;
- юридический/brand approval workflow.

### 7.6. Acceptance criteria

- Агент генерирует релевантный текст по продукту, каналу и сегменту.
- Текст появляется в communication activity текущего flow.
- Для продуктового сценария выбран существующий offer template.
- Пользователь может изменить тональность или выбрать другой вариант текста.

## 8. Автоматическая сегментация

### 8.1. Что прототипируем

Минимальный сценарий поиска или предложения целевого сегмента под продукт/кампанию. В текущем репозитории уже есть базовый демонстрационный контур сегментации: агент предлагает гипотезы, сопоставляет их с существующими Target Groups и маркирует новые сегменты как рекомендации без создания в prod.

### 8.2. Клиентские ожидания из заметок

- **СберМобайл**: интересен поиск новых микросегментов под продукт на основе GenAI.
- **ТТ Мобайл**: автоматическая сегментация входит в приоритеты после мониторинга и эмулятора.
- **UCell**: важна автоматическая сегментация, но в итоговом приоритете ниже песочницы, Copilot и агентов кампаний.

### 8.3. Опора на готовое

В репозитории уже есть демонстрационная реализация автоматической сегментации:

- `backend/agents/segment_agent.py` — LLM-backed агент сегментации, который получает справочник Target Groups, формирует 2–3 гипотезы аудитории, валидирует ответ и явно различает существующую Target Group и новый рекомендательный сегмент.
- `POST /api/segments/suggest` — backend endpoint для запроса сегментных гипотез из UI/Builder.
- Вкладка `Segments` в `frontend/src/components/FloatingWidget.tsx` — точка входа в сценарий сегментации из плавающего AI-widget.
- `frontend/src/components/SegmentPanel.tsx` — UI-панель, где пользователь описывает продукт/цель, получает гипотезы, видит предупреждения и может передать выбранный сегмент в Builder.

Текущий контур остается демонстрационным: он использует справочник Target Groups и mock/AdTarget данные, но не создает новые Target Groups в prod, не считает размер сегмента на реальной клиентской базе и не применяет contact policies в продуктивном контуре.

### 8.4. MVP-требования

1. Пользователь описывает продукт, цель кампании и базовые ограничения аудитории.
2. Агент предлагает 2–3 гипотезы сегментов для кампании.
3. MVP должен поддерживать отдельный demo-mode: **LLM собирает оптимальный сегмент из контактной базы / mock contact base**.
4. Для прототипа используется не prod-интеграция с клиентской DWH/CDP/AdTarget segmentation, а демонстрационная контактная база, mock contact base или агрегированный профиль признаков. Эти данные нужны только для объяснимого подбора сегмента в демо и не подтверждают реальный размер аудитории, наличие согласий или готовность сегмента к запуску.
5. Для каждой гипотезы агент должен указать:
   - название сегмента;
   - описание аудитории;
   - почему сегмент релевантен продукту;
   - предполагаемые признаки/условия отбора в человеческом виде;
   - риск или ограничение сегмента.
6. Для нового оптимального сегмента результат должен включать:
   - название нового сегмента;
   - краткий selling-friendly инсайт;
   - критерии отбора;
   - исключения;
   - ожидаемый эффект;
   - риск/ограничение;
   - пометку `Demo only / требует реализации в контуре сегментации`.
7. Если в справочнике уже есть подходящая Target Group, агент предлагает использовать ее.
8. Если готовой Target Group нет или новый сегмент лучше подходит для демо-кампании, агент формирует описание сегмента как рекомендацию, без автоматического создания в AdTarget.
9. Результат может быть передан в Builder как выбранная целевая аудитория или текстовое ограничение.

### 8.5. Границы MVP

Входит:

- генерация гипотез микросегментов под продукт;
- генерация нового сегмента из mock contact-base признаков или агрегированного профиля признаков;
- сопоставление с существующими Target Groups, если они доступны;
- текстовое описание критериев сегмента;
- передача выбранной гипотезы в сценарий проектирования кампании.

Не входит:

- реальное создание сегмента или новой Target Group в prod/AdTarget;
- расчет размера сегмента на реальных клиентах;
- применение контактных политик, согласий, opt-out, frequency cap и юридических ограничений в prod;
- ML-кластеризация клиентской базы;
- NBA/next best action на уровне абонента.

### 8.6. Acceptance criteria

- По описанию продукта агент предлагает 2–3 сегментные гипотезы.
- Для каждой гипотезы есть объяснение релевантности и критерии отбора.
- Агент предлагает существующую Target Group, если она явно подходит.
- Агент может предложить не только существующую Target Group, но и новый оптимальный сегмент с понятным объяснением, почему он лучше для демо-кампании.
- Новый сегмент явно помечен как `Demo only / требует реализации в контуре сегментации` и не выдается за созданный prod-сегмент.
- Агент не утверждает, что создал новый сегмент, если такой интеграции нет.

## 9. Сквозной demo-сценарий MVP

1. Пользователь в Copilot задает вопрос по CVM/документации и получает ответ с источниками.
2. В инициативе «Автоматическая сегментация» пользователь описывает продукт и получает 2–3 гипотезы аудитории.
3. В Builder пользователь выбирает гипотезу/существующую Target Group и генерирует кампанию.
4. В инициативе «Генерация контента» пользователь просит 2 варианта текста под продукт и сегмент; выбранный текст попадает во flow.
5. Кампания сохраняется как тестовая и появляется в реестре песочницы.
6. Пользователь запускает test run в эмуляторе и получает ошибки/рекомендации.
7. Агент мониторинга показывает pre-launch score и объясняет проблемы.
8. Агент оптимизации выдает 3–5 рекомендаций по параметрам кампании, без автоматического применения.
9. После подтвержденного запуска мониторинг показывает post-launch показатели и рекомендации.
